# 09 — API Contracts

**Status:** Draft
**Conventions:** REST-ish Route Handlers under `app/api/*`. JSON in, JSON out. All mutations from the UI go through Server Actions; Route Handlers exist for uploads, webhooks, refresh, and any future external client.

## Status codes

| Code | When |
|------|------|
| 200 | Successful read/update |
| 201 | Successful create |
| 204 | Successful delete (no body) |
| 400 | Validation error (Zod fail). Body: `{ error: string, fields?: {...} }` |
| 401 | No session. Body: `{ error: 'Unauthorized' }`. (Server Actions redirect to `/login` instead.) |
| 403 | Session but no permission. Body: `{ error: 'Forbidden' }` |
| 404 | Resource not found |
| 405 | Method not allowed |
| 409 | Conflict (e.g. duplicate unique constraint) |
| 413 | Payload too large |
| 422 | Semantically invalid (e.g. invalid state transition). Body: `{ error: string, code?: string }` |
| 429 | Rate limited. Body: `{ error: 'Too many requests', retryAfter: number }`. Header `Retry-After`. |
| 500 | Unhandled server error. Body: `{ error: 'Internal Server Error', requestId: string }`. |

`requestId` is generated by the proxy and echoed in logs + error responses for correlation.

## Authentication on routes

Every route handler starts with:

```ts
import { requireSession, requireRole, requireClientAccess } from '@/lib/dal/session'

export async function GET() {
  const claims = await requireSession()           // throws unauthorized() if no session
  // or
  await requireRole('SUPER_ADMIN')                // throws forbidden() if wrong role
  // or
  await requireClientAccess(clientId)             // IDOR protection
  // ...
}
```

`unauthorized()` and `forbidden()` automatically render the corresponding error pages when called from a Server Component context; from a Route Handler, they throw and Next returns 401/403.

Actually, the canonical pattern for Route Handlers is:

```ts
export async function GET() {
  let claims
  try {
    claims = await requireSession()
  } catch (err) {
    if (err instanceof UnauthorizedError) return NextResponse.json({ error: 'Unauthorized' }, { status: 401 })
    throw err
  }
}
```

Pick the explicit form when you need to control the response shape; pick the throw form when the default error page is fine. Document which.

## Endpoints

### Auth

| Method | Path | Body / Query | Response |
|--------|------|--------------|----------|
| POST | `/api/auth/refresh` | (none; reads cookie) | `{ ok: true }` + sets cookies |
| POST | `/api/auth/logout` | (none) | `{ ok: true }` + clears cookies |

`/login` and `/register` don't have route handlers — they're Server Actions.

### Admin

All require `requireRole('SUPER_ADMIN')`.

| Method | Path | Body / Query | Response |
|--------|------|--------------|----------|
| GET | `/api/admin/dashboard` | — | `{ stats, submissionsByStatus, recentActivity }` |
| GET | `/api/admin/clients?page=1&limit=20&q=` | — | `{ data: ClientDTO[], pagination }` |
| GET | `/api/admin/clients/[id]` | — | `{ client: ClientDetailDTO }` |
| POST | `/api/admin/invites` | `{ email, companyName, contactName }` | `201 { slug, inviteLink }` |
| GET | `/api/admin/invites` | — | `{ data: InviteDTO[] }` |
| DELETE | `/api/admin/invites/[slug]` | — | `204` |
| GET | `/api/admin/team` | — | `{ data: TeamMemberDTO[] }` |
| POST | `/api/admin/team` | `{ email, name, department }` | `201 { id }` |
| DELETE | `/api/admin/team/[id]` | — | `204` |
| GET | `/api/admin/forms` | — | `{ data: FormDTO[] }` |
| POST | `/api/admin/forms` | `{ title, description, formSchema }` | `201 { id }` |
| PUT | `/api/admin/forms/[id]` | `{ title?, description?, formSchema?, isActive? }` | `200 { form }` |
| GET | `/api/admin/audit?resource=&from=&to=` | — | `{ data: AuditLogDTO[], pagination }` |

Mutations from the admin UI go through Server Actions; these GETs cover dashboard analytics and list endpoints that benefit from HTTP caching.

### Team

All require `requireRole('TEAM_MEMBER')` (or `SUPER_ADMIN`).

| Method | Path | Notes |
|--------|------|-------|
| GET | `/api/team/dashboard` | Stats scoped to assigned clients |
| GET | `/api/team/clients` | Assigned clients only |

### Client

All require `requireRole('CLIENT')` and re-derive `clientId` from the session, **ignoring any `clientId` in the request**.

| Method | Path | Notes |
|--------|------|-------|
| GET | `/api/client/profile` | Current user's client profile |
| GET | `/api/client/forms` | Active forms |
| GET | `/api/client/submissions` | Own submissions |

### Comments

| Method | Path | Body | Notes |
|--------|------|------|-------|
| GET | `/api/comments?clientId=&submissionId=` | — | Filters internal based on viewer role |
| POST | `/api/comments` | `{ clientId, submissionId?, parentId?, message, isInternal? }` | Client cannot set `isInternal` |
| PATCH | `/api/comments/[id]` | `{ message }` | Author within 24h |
| DELETE | `/api/comments/[id]` | — | Author or admin |

### Uploads

| Method | Path | Body | Notes |
|--------|------|------|-------|
| POST | `/api/uploads` | multipart: `file`, `clientId`, `submissionId?` | Returns `201 { id, storageKey, mimeType, size }` |
| GET | `/api/uploads/[id]` | — | Streams the file (or redirects to presigned URL) |
| DELETE | `/api/uploads/[id]` | — | `204` |

### Webhooks

External systems only. Authenticated via signature header per integration.

| Method | Path | Notes |
|--------|------|-------|
| POST | `/api/webhooks/email` | Email bounce/delivery notifications from SMTP provider |
| POST | `/api/webhooks/cron/cleanup-invites` | Vercel Cron — call signature-verified |

### Utility

| Method | Path | Notes |
|--------|------|-------|
| GET | `/api/health` | `{ ok: true, time, db: 'up' \| 'down' }`. No auth. Used by uptime monitors. |

## Request and response envelope

### Success

```json
{ "data": <payload> }
```

Or, for paginated list endpoints:

```json
{
  "data": [ /* items */ ],
  "pagination": { "page": 1, "limit": 20, "total": 134 }
}
```

### Error

```json
{
  "error": "human-readable message",
  "code": "MACHINE_CODE",      // optional, for known errors
  "fields": { "email": ["..."] },  // optional, for validation
  "requestId": "req_..."        // always present on 5xx
}
```

## Versioning

No URL versioning (`/api/v1/...`) for v1. If we ever need breaking changes, we'll introduce `/api/v2/*` and freeze `/api/v1/*` for a deprecation window. No plans to do this in v1.

Non-breaking changes (adding fields, adding endpoints) don't require versioning.

## Rate limits

| Endpoint | Limit |
|----------|-------|
| `POST /api/auth/login` (via action) | 10/min per IP, 5/5min per email |
| `POST /api/auth/register` (via action) | 5/hour per IP, 3/hour per slug |
| `POST /api/uploads` | 50/hour per user |
| `POST /api/comments` | 30/min per user |
| All other authenticated | 300/min per user |
| All unauthenticated (excl. auth) | 100/min per IP |

Implementation: token-bucket via Upstash Redis (prod) or in-memory `Map` (dev). See `10-security.md`.

## Caching

Most endpoints are dynamic (per-user data). Caching strategy:

- `Cache-Control: private, no-store` on everything authenticated.
- `GET /api/health` is cacheable for 10 seconds.
- Server Component fetches use the Next 16 fetch cache where helpful; see `next-cache-components` skill for `use cache`.
